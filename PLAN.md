# Training Recorder → User Documentation Generator
## Подробный план реализации для Cursor Composer

> **Что это.** Chrome-расширение + бэкенд, который превращает голосовое обучение по веб-интерфейсу
> в готовую пользовательскую инструкцию (Markdown/HTML), со скриншотами стабильных состояний и
> голосовыми пояснениями.
>
> **Ключевая идея.** Это не запись кликов, это запись обучения.
> DOM и действия = факты. Голос = смысл. LLM = документ.

---

## 0. Как Cursor должен работать с этим документом

**ПРАВИЛА ВЫПОЛНЕНИЯ (читать обязательно):**

1. Идём **строго последовательно**: Эпик 0 → 1 → 2 → 3 → 4. Внутри эпика — по номерам задач.
2. Для каждой задачи цикл: **реализовать → прогнать «Тест приёмки» из задачи → если зелёный, перейти к следующей; если красный — чинить, не переходить дальше.**
3. **Не забегать вперёд.** Не реализовывать задачу N+1, пока не пройден тест задачи N. Зависимости указаны в каждой задаче.
4. **Контракт данных (Эпик 0, T0.1) — священен.** Все остальные задачи опираются на него. Если по ходу нужно поменять контракт — сначала правим схему и типы, потом всё, что от них зависит.
5. Каждая задача указывает: **цель, затрагиваемые файлы, начинку (что именно писать), тест приёмки.** Делать ровно в этих границах, не раздувать скоуп.
6. Стек фиксирован (см. раздел 2). Не менять без явной причины.
7. После каждого эпика — короткая проверка, что весь эпик собирается и проходит свои тесты, потом дальше.

**Definition of Done для всего MVP:** из записи 3–5 минут (запись вкладки + микрофон + действия) на выходе
получается отредактируемый документ-инструкция со скриншотами, который пользователь правит за разумное время,
а не переписывает с нуля.

## 1. Архитектурные решения (зафиксированы, не пересматривать в MVP)

| # | Решение | Почему |
|---|---------|--------|
| A | **MVP пишет ТОЛЬКО активную вкладку**, не «весь экран» | DOM-контекст доступен лишь во вкладке с content script. Запись экрана = потеря DOM = обычный Loom. Вся отстройка держится на DOM. |
| B | **Скриншоты = кадры из видеопотока (canvas), НЕ `chrome.captureVisibleTab`** | Снимает квоту (~2 кадра/сек), стоимость и невозможность снимать неактивную вкладку. Бонус: «заменить кадром из видео» — та же механика бесплатно. |
| C | **Микрофон пишется отдельной дорожкой/файлом** | Чистая транскрипция без звука вкладки. |
| D | **Единые часы `t0`** | Все события, аудио и видео таймштампятся относительно `t0`. Whisper отдаёт таймкоды от начала аудио = тот же `t0`. Без этого голос↔действия не выровнять. |
| E | **На бэкенд шлём audio + timeline.json + отобранные скрины. Видео — опционально/асинхронно** | 90% ценности в аудио+таймлайне+скринах. Сырое видео тяжёлое. |
| F | **Скрин ловится по СОСТОЯНИЮ объектов (сеть idle + DOM тишина + нет лоадеров + супер­седизация), НЕ по таймеру** | Фикс по времени ломается на лагах сети и быстрых переходах. Это ядро продукта (см. T1.4). |
| G | **Запись ведётся в offscreen document, не в service worker** | В MV3 service worker умирает и не держит `MediaRecorder`. |
| H | **Privacy — gate, а не «потом».** Маскирование password/PII на клиенте ДО отправки + экран-предупреждение + удаление артефактов | B2B-сегменты работают с боевыми данными. Без этого не продать. |
| I | **Vision — только fallback** | DOM точнее, дешевле, быстрее. Vision вызывается лишь когда у элемента нет текста/роли или скрин `low_confidence`. |

```
│   │   ├── render.py               # MD/HTML (T2.6)
│   │   └── vision.py               # fallback (T4.1)
│   ├── prompts/
│   │   ├── generate_doc.md         # промпт генерации (T2.4)
│   │   └── vision_describe.md      # промпт vision (T4.1)

│   ├── models.py                   # ORM/pydantic
│   └── requirements.txt
└── editor/                         # ОТДЕЛЬНОЕ веб-приложение, раздаётся бэкендом, НЕ часть расширения (T3.x)
    └── (React+Vite, открывается в обычной вкладке по ссылке от расширения)
```

**Стек:**
- Extension: Manifest V3, **ванильный JS без сборщика** (popup, content scripts, service worker, offscreen). Это стандартный путь для расширений и снимает CSP-боль MV3 (запрет `eval`/удалённого кода). offscreen API, `getDisplayMedia({preferCurrentTab:true})` или `chrome.tabCapture`, `getUserMedia` для микрофона.
- Backend: Python 3.11+, FastAPI, SQLite (через SQLAlchemy или sqlite3), хранилище — локальная папка `storage/` (позже S3).
- AI: OpenAI Whisper (`whisper-1`, `verbose_json`) для транскрипции; Claude/GPT для генерации; vision-модель для fallback.
- Рендер: Markdown → HTML (markdown-it/markdown2); PDF на MVP = печать HTML браузером.
- Редактор: **отдельное веб-приложение (React+Vite), которое раздаёт бэкенд и которое открывается в обычной вкладке браузера.** Это НЕ код расширения — поэтому ограничения MV3 (CSP, запрет `eval`) на него не действуют, React+Vite работает как на обычном сайте. Расширение после записи лишь открывает ссылку `editor/recording/{id}`.

> **Важно про границы:** «расширение» = ванильный JS внутри Chrome (Эпики 0–1 + точки интеграции). «Редактор» = обычный сайт на бэкенде (Эпик 3). Не путать и не пытаться собрать React внутрь расширения.

> **Если в репо уже есть `extension/` и `backend/main.py` от QuickLoom** — переиспользовать каркас записи и эндпойнты,
> приводя к этой структуре, а не писать с нуля.

# ЭПИК 0. Контракт и каркас

## T0.1 — Единый контракт таймлайна
**Зависит от:** ничего. **Делать первым.**

**Цель.** Зафиксировать формат данных, общий для расширения, бэкенда и промптов.

**Файлы:** `shared/timeline.schema.json`, `shared/timeline.types.ts`.

**Начинка — описать сущности:**

```ts
// Всё время — миллисекунды от t0 (момент старта записи).
interface RecordingMeta {
  recordingId: string;
  t0: number;            // Date.now() в момент старта (абсолютный, для справки)
  url: string;           // стартовый URL
  title: string;
  durationMs: number;
  userAgent: string;
}

type EventType =
  | "click" | "input" | "submit" | "navigation"
  | "modal_open" | "menu_select" | "focus" | "keypress_enter";

interface ElementContext {
  role: string | null;        // aria role или вычисленная
  text: string | null;        // видимый текст / aria-label (маскированный)
  placeholder: string | null;
  label: string | null;       // <label>, aria-labelledby, ближайшая подпись
  nearbyText: string | null;  // текст ближайшего значимого контейнера
  tag: string;
  cssPath: string;            // стабильный селектор для отладки
  bbox: { x: number; y: number; w: number; h: number }; // в координатах видимой области
  masked: boolean;            // было ли что-то скрыто маскированием
}

interface RecEvent {
  id: string;
  type: EventType;
  ts: number;                 // от t0
  url: string;
  target: ElementContext | null;
  value?: string | null;      // для input — уже маскированный/обрезанный
}

interface Screenshot {
  id: string;
  ts: number;                 // от t0
  eventId: string | null;     // действие, к которому привязан result state
  confidence: "high" | "low"; // low = стабилизацию не поймали
  width: number; height: number;
  // путь к файлу/блобу проставляется при сохранении
  candidates?: string[];      // id альтернативных кадров вокруг точки
}

interface TranscriptSegment {
  start: number; end: number; // от t0
  text: string;
  words?: { word: string; start: number; end: number }[];
}

// Выход LLM:
interface Step {
  id: string;
  title: string;              // короткий заголовок шага
  body: string;              // пояснение (очищенная речь + смысл)
  screenshotId: string | null;
  eventIds: string[];        // какие действия схлопнулись в шаг
  needsReview: boolean;      // данных не хватило — пометка проверить
}

interface GeneratedDoc {
  title: string;
  purpose: string;           // назначение
  audience: string;          // кому
  prerequisites: string;     // что нужно до начала
  steps: Step[];
  warnings: string[];        // частые ошибки / предупреждения
  result: string;            // ожидаемый результат
}
```

**Тест приёмки:**
- В `shared/` есть схема + типы; пример валидного `timeline.json` (мок) проходит валидацию JSON-schema.
- TS компилируется без ошибок.

## T0.2 — Каркас MV3 + offscreen-запись
**Зависит от:** T0.1.

**Цель.** Расширение записывает активную вкладку из offscreen document, не из service worker.

**Файлы:** `extension/manifest.json`, `service-worker.js`, `offscreen.html`, `offscreen.js`, `popup/*`.

**Начинка:**
- `manifest.json`: MV3, permissions: `offscreen`, `activeTab`, `scripting`, `storage`, `tabs`; `host_permissions: ["<all_urls>"]`.
- `service-worker.js`: по клику в popup создаёт offscreen document (`chrome.offscreen.createDocument`, reason `USER_MEDIA`/`DISPLAY_MEDIA`), шлёт ему команды start/stop, ретранслирует результат.
- `offscreen.js`: держит `MediaRecorder`, собирает чанки, по stop отдаёт blob(ы) обратно.
- Фиксируем `t0 = Date.now()` в момент фактического старта и кладём в `RecordingMeta`.

**Тест приёмки:**
- Клик «Запись» → идёт захват активной вкладки из offscreen → «Стоп» → скачивается `.webm` ненулевого размера.
- `t0` сохранён в meta.

# ЭПИК 1. Захват в расширении

## T1.1 — Запись вкладки + отдельная дорожка микрофона
**Зависит от:** T0.2.

**Цель.** Два независимых потока с общим `t0`.

**Файлы:** `offscreen.js`, `service-worker.js`.

**Начинка:**
- Видео вкладки: `getDisplayMedia({preferCurrentTab:true, video:true})` (или `chrome.tabCapture.capture`). Это поток для видео и для вырезки кадров (T1.4).
- Микрофон: отдельный `getUserMedia({audio:true})` → отдельный `MediaRecorder` → отдельный `audio.webm`.
- Оба рекордера стартуют после фиксации `t0`; запоминаем смещение старта каждого относительно `t0` (в идеале ~0).
- На выходе: `video.webm`, `mic.webm`, `meta.json`.

**Тест приёмки:**
- На выходе три артефакта. `mic.webm` содержит только голос (без звука страницы). Длительности совпадают ±300мс.

## T1.2 — Content script: сбор событий + DOM-контекст
**Зависит от:** T0.1, T1.1.

**Цель.** Логировать действия пользователя с контекстом элемента в формате `RecEvent`.

**Файлы:** `content/content.js`.

**Начинка:**
- Слушатели: `click`, `input` (debounce), `submit`, `focusin`, `keydown[Enter]`; навигация через `history.pushState`/`popstate`/`hashchange` + смена `location`; открытие модалок — эвристика (появление видимого элемента `role=dialog`/`aria-modal=true`).
- Для каждого события собрать `ElementContext`:
  - `text`: `innerText`/`aria-label`/`value` кнопки (обрезать до ~120 симв).
  - `label`: связанный `<label for>`, `aria-labelledby`, или ближайшая текстовая нода слева/сверху.
  - `nearbyText`: текст ближайшего значимого предка (карточка/секция).
  - `role`: `getAttribute('role')` или эвристика по тегу.
  - `cssPath`: устойчивый селектор (nth-child путь).
  - `bbox`: `getBoundingClientRect` в координатах видимой области (для аннотаций и кропа).
- Каждому событию — `ts = Date.now() - t0`, уникальный `id`.
- События буферизуются и отдаются в service worker / складываются для финального `timeline.json`.

**Тест приёмки:**
- На тестовой странице (форма + меню + модалка) каждый клик/ввод/сабмит даёт корректный `RecEvent` с заполненным `target.text`/`label`/`bbox` и валидным `ts`.

## T1.3 — Маскирование на клиенте (privacy gate)
**Зависит от:** T1.2.

**Цель.** Чувствительные данные не попадают в `timeline.json` и не уходят на бэкенд.

**Файлы:** `content/masking.js`.

**Начинка:**
- Любой `input[type=password]` → `value` и `text` заменяются на `••••`, `masked=true`.
- Эвристики PII (по `name`/`autocomplete`/`type`): `email`, `tel`, `cc-number`, `iban`, `inn`/`снилс`-паттерны — значения маскируются/обрезаются.
- Элементы с атрибутом `data-sensitive` или внутри контейнера `[data-sensitive]` — текст исключается.
- Глобальный лимит длины `value` (например 200 симв), чтобы не утащить полотна текста.

**Тест приёмки:**
- В `timeline.json` после прохода по форме с паролем/почтой/телефоном нет реальных значений; стоит `masked=true`. Юнит-проверка маскирующих функций зелёная.

## T1.4 — Детектор стабилизации + отбор кадров (ЯДРО ПРОДУКТА)
**Зависит от:** T1.1, T1.2.

**Цель.** Скрин снимается по **состоянию объектов**, а не по таймеру. Устойчив к лагам сети и быстрым переходам.

**Файлы:** `content/stabilizer.js`, `content/net-hook.js`, `lib/frame-capture.js`.

**Начинка — конечный автомат на каждое «значимое» действие:**

Значимые действия (триггеры кандидата на скрин): `navigation`, `submit`, `menu_select`, `modal_open`,
клик по элементу с `role=button`/ссылке. Обычный `input` скрин не триггерит.

```
Состояния кандидата: PENDING → WAITING → CAPTURED | SUPERSEDED | TIMED_OUT

on significant_action(eventId):
    if есть активный кандидат в WAITING:
        пометить его SUPERSEDED   // быстрый второй переход — старый кадр выбрасываем
    создать кандидат(eventId), state = WAITING
    запустить проверку стабилизации

Стабилизация считается достигнутой, когда ОДНОВРЕМЕННО держится QUIET_WINDOW (≈400мс):
    1) network idle:  активных fetch/XHR == 0
    2) DOM тишина:    нет значимых мутаций за окно
    3) нет лоадеров:  нет видимого [aria-busy=true], [role=progressbar],
                      .*spinner* / .*skeleton* / .*loading*, типовых фреймворк-лоадеров
    4) лэйаут стабилен: scrollHeight/размер main-контейнера не меняется между двумя замерами

если все 4 держатся непрерывно QUIET_WINDOW:
    state = CAPTURED → вырезать кадр(ы) из видеопотока, confidence=high
если прошло MAX_WAIT (≈8000мс) и не поймали:
    state = TIMED_OUT → взять лучший доступный кадр, confidence=low
```

**Перехват сети (`net-hook.js`):** инжектируется в контекст страницы (через `<script>` инъекцию, т.к. content script
изолирован от `window.fetch` страницы). Монки-патчим `window.fetch` и `XMLHttpRequest` → счётчик активных запросов,
транслируем счётчик в content script через `postMessage`/CustomEvent.

**MutationObserver:** на `document.body`, `subtree:true`. Считаем значимыми: добавление/удаление видимых узлов,
изменения атрибутов размеров/видимости. Игнор: смена `style` курсора, частые текстовые тики таймеров, изменения вне вьюпорта.

**Вырезка кадров (`frame-capture.js`):** из `MediaStream` видео → `<video>` (или `ImageCapture`) → `<canvas>` → `toBlob('image/jpeg', 0.85)`.
В момент CAPTURED берём **3 кадра** (в точке стабилизации и ±120мс) → основной + `candidates[]` для редактора.

**Параметры (вынести в конфиг, чтобы крутить):** `QUIET_WINDOW=400`, `MAX_WAIT=8000`, `LAYOUT_SAMPLE=150`.

**Тест приёмки (критичный — проверять руками на реальных кейсах):**
1. **Медленная загрузка:** на странице с искусственной задержкой 2 сек после клика скрин снимается **после** появления контента, не на белом экране/скелетоне.
2. **Быстрый второй переход:** клик A, через 200мс клик B → кандидат A помечен `SUPERSEDED`, в итоге есть скрин только стабильного состояния B (не промежуточный мусор).
3. **Лоадер:** пока виден спиннер — скрин не снимается; снимается после его исчезновения.
4. **Таймаут:** на странице, которая «никогда не успокаивается» (постоянные мутации), через MAX_WAIT появляется кадр с `confidence=low`.
5. На выходе у скрина есть `candidates[]` из нескольких кадров.

## T1.5 — Экран-предупреждение + удаление артефактов
**Зависит от:** T1.1.

**Цель.** Доверие: пользователь видит, что уходит на сервер, и может всё удалить.

**Файлы:** `popup/popup.html`, `popup/popup.js`.

**Начинка:**
- Перед стартом — экран со списком: «Записывается: видео вкладки, ваш голос, действия и тексты элементов. Пароли и помеченные поля скрываются. Данные отправляются на сервер для генерации инструкции (транскрипция через OpenAI Whisper).»
- Чекбокс согласия (без него старт заблокирован).
- После записи — кнопка «Удалить запись и все артефакты» (чистит локальное хранилище/блобы; если уже загружено — шлёт `DELETE /recording/{id}`).

**Тест приёмки:**
- Без согласия старт недоступен. Кнопка удаления реально стирает локальные артефакты (и дёргает DELETE, если был аплоад).

# ЭПИК 2. Бэкенд и AI

## T2.1 — Эндпойнт приёма артефактов
**Зависит от:** T0.1.

**Цель.** Принять и сохранить артефакты, вернуть `recordingId`/`jobId`.

**Файлы:** `backend/routes/process.py`, `backend/storage.py`, `backend/models.py`, `backend/main.py`, `backend/requirements.txt`.

**Начинка:**
- `POST /process` (multipart): `mic.webm`, `timeline.json`, `screenshots[]` (jpeg). Видео — опциональное поле, можно асинхронно/позже.
- Валидация `timeline.json` по схеме T0.1.
- Сохранение в `storage/{recordingId}/...`; запись в SQLite (`recordings`, `jobs`).
- Возврат `{recordingId, jobId, status:"received"}`.
- `DELETE /recording/{id}` — удаляет артефакты (для T1.5).

**Тест приёмки:**
- `curl` с мок-артефактами → 200 + `recordingId`. Файлы на диске, строка в БД. `DELETE` чистит.

## T2.2 — Транскрипция (Whisper verbose_json)
**Зависит от:** T2.1.

**Цель.** Получить `TranscriptSegment[]` с пословными таймкодами в едином clock.

**Файлы:** `backend/services/transcription.py`.

**Начинка:**
- Whisper `whisper-1`, `response_format="verbose_json"`, `timestamp_granularities=["word","segment"]`.
- Маппинг сегментов/слов в `TranscriptSegment` (время от t0 = время от начала аудио, т.к. mic стартовал на t0).
- Язык — авто; хранить как есть.

**Тест приёмки:**
- На тестовом `mic.webm` (10–15 сек речи) возвращается массив сегментов с `start/end/text` и словами; времена монотонны.

---

## T2.3 — Сборка единого таймлайна + первичная группировка
**Зависит от:** T2.2, T1.2/T1.4 (формат).

**Цель.** Слить транскрипт + события + скрины по `ts` и предварительно нарезать на шаги вокруг реплик.

**Файлы:** `backend/services/merge.py`.

**Начинка:**
- Объединить три потока в один отсортированный по `ts` список записей.
- **Якоря-реплики:** каждый связный сегмент речи (или пауза > N сек как граница) = смысловой якорь.
- Действия, попадающие в окно между двумя якорями, относятся к одному предварительному шагу.
- Каждому предварительному шагу сопоставить скрин(ы) ближайшего по времени стабильного `result state` (high-confidence приоритетнее low).
- На выход — `merged_timeline` (это вход для LLM, НЕ сырьё).

**Тест приёмки:**
- На мок-данных (реплики + клики + скрины) получается упорядоченный merged-timeline; действия корректно сгруппированы вокруг реплик; каждому предварительному шагу привязан скрин.

---

## T2.4 — LLM-генерация документа
**Зависит от:** T2.3.

**Цель.** Из `merged_timeline` сделать `GeneratedDoc`.

**Файлы:** `backend/services/generate.py`, `backend/prompts/generate_doc.md`.

**Начинка промпта (`generate_doc.md`):**
- Вход: расшифровка с таймкодами + список действий с контекстом элементов + список скринов + URL/заголовки.
- Инструкции модели:
  1. Понять цель процесса (из голоса).
  2. Разбить на **логические** шаги (не по каждому клику; группировать связанные действия).
  3. Очистить устную речь (убрать слова-паразиты), но **сохранить пояснения и нюансы**.
  4. Дать каждому шагу понятный заголовок.
  5. Привязать к каждому шагу правильный скрин (по eventId/ts из merged_timeline).
  6. Вынести отдельно: назначение, аудиторию, prerequisites, частые ошибки/предупреждения, ожидаемый результат.
  7. **Не выдумывать действий, которых не было.** Если данных не хватает — `needsReview=true`.
- Выход: строго JSON по схеме `GeneratedDoc` (T0.1). Использовать structured output / json mode.

**Тест приёмки:**
- На эталонном merged-timeline (сценарий «создание клиента» из ТЗ) возвращается валидный `GeneratedDoc`: 4–6 логических шагов, заполнены purpose/audience/warnings/result, есть привязки скринов, нет выдуманных шагов.

---

## T2.5 — Привязка шаг ↔ скрин (уточнение)
**Зависит от:** T2.4.

**Цель.** Гарантировать корректный основной скрин + альтернативы у каждого шага.

**Файлы:** `backend/services/screenshot_match.py`.

**Начинка:**
- Для каждого `Step.eventIds` найти result-state скрин (по `Screenshot.eventId`), приоритет high-confidence.
- Если у шага нет скрина — взять ближайший по `ts`; пометить `needsReview`.
- Приложить `candidates[]` (для «выбрать другой кадр» в редакторе).

**Тест приёмки:**
- Каждый шаг имеет `screenshotId` либо `needsReview=true`; альтернативы доступны.

---

## T2.6 — Рендер Markdown + HTML
**Зависит от:** T2.4.

**Цель.** Превратить `GeneratedDoc` в Markdown и HTML.

**Файлы:** `backend/services/render.py`.

**Начинка:**
- Структура по ТЗ: заголовок → назначение → кому → перед началом → шаги (заголовок + текст + скрин) → частые ошибки → результат.
- Markdown: скрины как `![](path)`. HTML: самодостаточный (скрины inline/относительные), пригодный для печати в PDF браузером.
- Эндпойнт `GET /recording/{id}/doc?format=md|html`.

**Тест приёмки:**
- `GET ...?format=md` и `?format=html` возвращают документ нужной структуры со скринами под шагами. HTML открывается и печатается в PDF.

`POST /process` → транскрипция → merge → LLM → привязка скринов → MD/HTML. Сквозной прогон на мок-записи даёт документ.

# ЭПИК 3. Редактор и экспорт

## T3.1 — Экран-редактор (отдельное веб-приложение, НЕ часть расширения)
**Зависит от:** T2.4–T2.6.

**Цель.** Дать пользователю быстро поправить результат перед экспортом — в обычной вкладке браузера.

**Файлы:** `editor/` (React+Vite); + раздача из `backend` (статика или dev-прокси); + точка интеграции в расширении.

**Начинка:**
- Это **обычный SPA**, который бэкенд отдаёт по `GET /recording/{id}` (HTML+ассеты). Открывается в новой вкладке, не внутри расширения → ограничения MV3 на него не распространяются.
- Расширение после успешного `POST /process` делает `chrome.tabs.create({url: editorUrl})` с `recordingId`.
- Тянет `GeneratedDoc` по `recordingId` (`GET /recording/{id}/doc`).
- Список шагов: заголовок, текст (редактируемые), привязанный скрин.
- Действия: удалить шаг, объединить два шага, заменить скрин (выбрать из `candidates[]` или другой кадр), редактировать текст.
- Шаги с `needsReview=true` подсвечены.
- Сохранение: `PUT /recording/{id}/doc` (обновлённый `GeneratedDoc`).

**Тест приёмки:**
- После записи расширение открывает редактор в новой вкладке с нужным `recordingId`.
- Можно отредактировать текст, удалить/объединить шаги, сменить скрин; изменения сохраняются и переживают перезагрузку.

**Тест приёмки:**

# ЭПИК 4. Vision-fallback (минимально, последним)

## T4.1 — Vision только как fallback
**Зависит от:** Эпик 2.

**Цель.** Закрыть пробелы DOM, не делая vision основой.

**Файлы:** `backend/services/vision.py`, `backend/prompts/vision_describe.md`.

**Начинка:**
- Триггеры вызова: у элемента нет `text`/`role`/`label`, ИЛИ скрин `confidence=low` и непонятно состояние.
- Один vision-запрос: «опиши, что это за элемент/состояние экрана» → результат подмешивается в контекст шага.

**Тест приёмки:**
- Для элемента-иконки без текста vision возвращает описание; на нормальных элементах vision **не вызывается**; счётчик вызовов в пределах лимита.

```
E0: T0.1 → T0.2
```
